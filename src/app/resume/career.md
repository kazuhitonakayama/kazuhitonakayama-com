# 職務経歴書

## 職務要約

ヴィーガン領域のスタートアップ「ブイクック」にてテックリードとして、レシピ投稿サイト・EC・ネットスーパーなど複数事業の立ち上げに従事。前身ECの構造的な課題を解消するためネットスーパー業態へのピボットを技術側からリードし、構想から約1ヶ月でリリースを実現。少人数・低稼働の体制下で、技術選定からスコープ設計、ステークホルダー間の合意形成までを担った。

その後、GMOペパボ株式会社にて日本最大のハンドメイドマーケットプレイス「minne」のプロダクト開発エンジニアとして、バックエンド・フロントエンド・インフラを横断的に担当。検索基盤刷新・認証強化・決済導入など事業インパクトの大きい開発を主導しながら、AI活用による組織全体の開発生産性向上やスクラム改善にも取り組んできた。

「作ること」ではなく「事業を前に進めること」を判断軸に、技術と事業・組織の橋渡しを一貫して担ってきた。

## 職務経歴

### 株式会社ブイクック（2022年3月〜2023年3月 / テックリード / プロジェクトリード）

#### 関わったサービス

- ヴィーガンレシピ投稿サイト「vcook」
- ヴィーガン商品専門ECサイト「vcookmall」
- ヴィーガン商品専門ネットスーパー「ブイクックスーパー」

#### 役割・責任範囲

技術で事業を成長させることをミッションとし、テックリードとして技術選定・コードレビュー・コード品質担保に責任を持つ。バックエンド／フロントエンドを横断した実装も並行して担当。ブイクックスーパーでは、少人数・低稼働の開発体制を率いるプロジェクトリードとして、技術と事業の橋渡しを担った。

#### 事業課題を起点としたピボット — ブイクックスーパーの立ち上げ *[1]

前身のECサイト「vcookmall」では、複数メーカーの商品をまとめて購入する構造上、**メーカーごとに送料が発生し購入ハードルが高くなる**という事業課題を抱えていた。

この課題を解消するため、自社で在庫を持ち送料を一括管理できる**ネットスーパー業態へのピボット**を意思決定。新サービス「ブイクックスーパー」の立ち上げを技術側のリードとして牽引した。

- **「仮説検証スピードの最大化」を判断軸**に据え、Railsフルスクラッチ（3〜5ヶ月想定）ではなく Shopify を採用。「作りたい技術」ではなく**「事業フェーズに最適な技術」を選ぶ意思決定**をチーム内で合意形成した
- 週2稼働のエンジニア2名・週1稼働のデザイナー2名＋CEO/COOという少人数・低稼働体制をリードし、デザインと開発の並走・職能を越えたコラボレーションを設計。構想から**約1ヶ月（実装24日間）でリリース**を達成
- 「スコープ / 期日 / 機能 / 予算」の優先順位をプロジェクト全体で言語化・共有し、**「何を作らないか」をチームで合意し続ける**スコープマネジメントを継続的に実施
- 週2回の定例で懸念事項・進捗・意思決定事項を共有し、低稼働メンバーでも判断に必要な情報が常に揃っている状態を維持

#### マネジメント観点の学び

- **オペレーションを含めたE2Eで品質を捉える視点**: 配送フローの実地検証まで踏み込めずリリース当日に運用混乱が発生。以降、人間のオペレーションを含めたE2Eの検証計画を要件定義段階から設計するように改めた
- **超短期開発における期待値マネジメント**: 技術的な正しさよりも事業フェーズ上の優先順位をステークホルダー間で継続的に翻訳・合意形成する役割の重要性を実感
- **「コードを書かない」選択を組織として肯定する難しさ**: 意思決定の背景を丁寧に言語化し続けることで、「事業を前に進めること」をチームの共通言語にする組織づくりの重要性を学んだ

### GMOペパボ株式会社（2023年4月〜現在 / minne プロダクト開発エンジニア / リーダー）

日本最大のハンドメイドマーケットプレイス「minne」にて、バックエンド・フロントエンド・インフラを横断的に担当。

#### 役割・責任範囲

Android（2名）・iOS（2名）・Web（約9名・配下に3チーム）で構成される開発組織において、各チームの**開発計画の順守**に責任を持つリーダーとして率先垂範。また、**セキュリティ向上の責任者**として、向こう1年間のセキュリティ対策ロードマップの策定・役割分担・実行への落とし込みまでを主導。

#### 1. 検索エンジン基盤の構築・移行

minneの**1,800万点超の作品**に対する検索体験の改善を、OpenSearch導入から外部検索エンジン S4 への移行まで一貫してリード。 *[2]

**OpenSearchベースの検索機能構築**

- 作品管理画面にOpenSearchベースのキーワード検索機能を実装。公開・限定公開・非公開の全ステータスを単一インデックスで扱う設計判断を行い、購入者向けと作家向けで異なるフィルタリングを実装。full reindex を回避する Update Mapping API 活用や Stats API によるキャパシティプランニングを経て、**インシデントゼロで完遂**
- 「トレンド性」に基づく新しいソートアルゴリズムを設計・実装し、直近の売れ筋作品を検索上位に表示するロジックを導入
- Elasticsearch のマッピング更新時に partial reindex ではマルチフィールド追加が反映されない問題を調査・解決し、再インデックス戦略の知見を社内外に発信 *[3]

**外部検索エンジン S4 への移行**

1,800万点規模での検索関連性が課題である一方、社内にOpenSearchを継続的にメンテナンスするリソースが不足。**「検索精度向上」と「運用負荷軽減」を同時に解消する手段**として外部検索エンジン導入を意思決定。

- **オフライン評価プロセスの設計** — 検証環境にminneの作品データを登録し、検索結果1〜3ページ目の各48点を「クリックしたくなるか」で⭕/❌評価する手動スコアリングを実施。本番投入前に上位ページの品質改善を定量検証
- **CDCを活用した差分更新パイプラインを当初見積もりの約半分（1ヶ月）で構築**。Debezium → Pub/Sub → GCF → SQS → Lambda のアーキテクチャを設計し、通常更新と大量更新の2系統パイプラインで安定稼働を実現
- **多腕バンディットによる自動ABテスト**を運用に組み込み、本番投入後も継続的に勝ちアルゴリズムへ露出を寄せる仕組みを構築

**成果**
- 多腕バンディットが外部検索エンジンを勝者と判定し、**現在は検索トラフィックの100%が外部検索エンジン経由**
- 移行プロジェクトは**インシデントゼロ**で完遂、当初見積もりの半分の期間で稼働
- OpenSearch の運用負荷を外部サービスへ委譲し、**社内リソースをプロダクト改善に集中できる構成**を実現

#### 2. セキュリティ強化 — パスワードレスログインの実現

パスワードリスト攻撃により**1日20件以上のアカウント乗っ取り**が発生。スパムメッセージを介した機密情報詐取や、それに伴うCS対応負荷の急増がサービス信頼性を脅かしていた。

バックエンド2名で仕様策定・ADR作成を行い、iOS/Android各1名への仕様伝達と実装リード、社内セキュリティ担当との合意形成を担当。

- **「100万人以上のユーザーを強制ログアウトさせない」を判断軸**に、パスワード強化ではなくパスワードレス化を選択。導入時点以前のアクセスユーザーにのみアクセストークンを失効させる設計で、既存ユーザー体験への影響を最小化
- reCAPTCHA v3をログイン・メッセージ送信の2エンドポイントに**1営業日以内**に導入し初動で攻撃量を抑制
- マジックリンクによるパスワードレスログインを**約1ヶ月で実現**
- AWS WAFで防御レイヤを構築 — (1) ログインリンク送信エンドポイントへの大量試行をルールベースで遮断、(2) 運用用に残したパスワードログイン画面は許可IP経由のみアクセス可能とする構成を設計
- モバイルアプリ向けGraphQLエンドポイントを提供し、全プラットフォームでのパスワードレス化を推進
- **13億レコードのアクセストークン管理テーブル**に対し計画停止2時間でメンテナンスを実施。revoke済みレコード削除クエリ向けインデックスを追加し、将来のデータ削除基盤を確立

**成果**
- **アカウント乗っ取り件数がゼロに**（導入前: 1日20件以上）
- **事実上すべてのユーザーがパスワードレスログインに移行**
- メンテは計画時間内に完遂、ユーザー影響なし

**継続改善**
導入後「マジックリンクメールが届かない」という到達性起因の問い合わせが発生。単一のログイン手段への依存リスクを学びとし、代替手段として**パスキー導入を推進中**。

#### 3. Eコマース機能開発

**Amazon Pay 1ステップ購入**
- 作品詳細画面から実質1ステップで購入完了できるAmazon Pay決済を導入
- iOS / Android / Web の3プラットフォーム横断でGraphQL APIの全体設計を主導
- **4チーム（Web、iOS、Android、業務委託）**を巻き込んだ開発体制を構築し期日通りにリリース
- **1日200件以上の恒常的な利用**を実現

**CSV一括発送通知機能**
- AWS Lambda / SQS / S3 / Shoryuken を活用したクラウドネイティブなアーキテクチャを設計・実装
- **211件の発送通知処理を従来の約105分から3秒に短縮**（最大1,000件対応）
- 大規模作家の業務負荷を大幅に軽減

**メッセージ詳細画面のリニューアル**
- Rails → Next.js へのフロントエンド刷新とバックエンドのGraphQL化を**約1ヶ月で完遂**
- minne PLUS（有料会員プラン）のメッセージテンプレート機能の土台を構築
- UNLEASHによるフィーチャーフラグを活用した段階的リリース戦略を設計・実行

#### 4. ゲーミフィケーション機能の立ち上げ — minneリワード

GMV減少による売上低下を事業課題として、(1)**新たな広告収益源の確保**、(2)**「推し作家への応援」ニーズ仮説の検証**、(3)**DAU増加**を狙い企画。累積ミッションによりminne上のユーザー活動を資産化し、**minneは広告収益・ユーザーはコインを得るWin-Winモデル**を設計。

4名チーム（バックエンド2名・iOS 1名・Android 1名 + ディレクター）のバックエンド側で設計主導。要件定義はディレクターと、アーキテクチャはバックエンド2名で確定させた。**「集計基盤をバックエンドに寄せる」提案を自ら発起**し、合意形成をリード。約3〜4週間で立ち上げ。 *[5]

- **「クライアント工数ゼロ」を判断軸とした設計**: 「ミッション追加のたびにiOS/Androidのリリースが必要」となる初期案（クライアント直接呼び出し方式）を却下し、ミッション定義・判定・付与をすべてバックエンドへ集約。**新規ミッション追加時にクライアントの工数がほぼ不要**となる構造を実現
- **マルチクラウドアーキテクチャの設計**: メインアプリへの変更を最小化するため、既存ログパイプライン（Rails → fluentd → Cloud Pub/Sub → Firestore）を実績集計に活用しつつ、ミッション進捗管理は AWS（Rails → SQS → Lambda → DynamoDB）で構築する**マルチクラウド構成**を採用。SQS非同期パイプラインによりレイテンシスパイクとスロットリングからメインアプリを保護
- **ホットパーティション対策**: Firestore は `missions/{user_id}/dailyProgress/{YYYYMMDD}` のユーザー起点ドキュメント構造を採用し、日付起点設計での書き込み集中を回避。DynamoDB ではGSIに `{YYYY-MM-DD}#{user_id % 10}` のシャードサフィックスを付与し10シャードに分散、読取時は ScatterGather で対応
- **不正防止のための2フェーズコイン付与**: 動画広告視聴ミッションでは、クライアント報告で仮付与（PENDING）→ 翌日に広告配信元の CSV レポートと突合 → 本付与する2段階フローを設計。**AWS Step Functions** でバッチ処理をオーケストレーション
- **GraphQL における Interface 型分離による認可設計**: 認証ユーザー向け `OwnMission`（進捗情報を含む）と非ログイン向け `GuestMission`（定義情報のみ）を型レベルで分離し、フィールド乖離への耐性と認可の型安全性を確保
- **SSoT 原則の遵守**: 累積ミッション（購入回数・応援スタンプ等）は新規データストアを作らず、RDB・DynamoDB の既存データから直接集計する判断を行い、データの出自に応じたストア選定を徹底

**成果**
- **1日あたり約4万円・年換算で約1,400万円の新たな広告収益源**を創出
- **ミッション追加工数が約1日で完結** — バックエンド側での集計基盤追加・ミッションレコード作成と、モバイル側のAPIリクエスト追加のみで完了。新規ミッション追加時にクライアントの追加リリースが不要という**スケーラブルな運用構造**が実運用で検証された

#### 5. インフラ・DevOps

- **Argo Rollouts による Progressive Delivery**: Kubernetes上での自動ロールバック機構を構築
- **本番リリースフローのECR化**: コンテナレジストリ障害によるリリース不能状態を解消。pod起動高速化とイメージ取得コスト削減も実現
- **CI/CDパイプラインの継続的改善**: dependabot PRの自動approve、Docker buildリトライ、デプロイ通知自動化など

#### 6. AI活用による開発生産性向上

チーム規模40名に拡大する中で、PR数増加に伴うレビュー負荷の偏在が開発のボトルネックになっていた。またエンジニア以外の職種（CS・デザイナー・マネージャー）が簡単な改修や仕様調査を自己完結できれば、エンジニアの稼働を本質課題に集中できるという仮説を持っていた。

「今のチームに必要」と判断し自ら起案。**Bedrock基盤の導入・主要リポジトリへの展開・成果発表・他部署へのナレッジ展開まで一貫して意思決定者として主導**し、**約2週間で全社展開可能な基盤を構築**。

- minneの**主要リポジトリ全て（バックエンド・フロントエンド・iOS・Android・プラットフォーム）**に Claude Code Actions を導入
- **Figma MCP** の導入によりデザインシステムに基づく一貫した実装を可能に
- **AIによるPRサイズ自動分類（影響範囲ベースの5段階判定）・コードレビュー・条件付きAuto Merge**の仕組みを構築。**「AI活用で増えたPRがレビュー負荷を再生産する副作用」をAI自身で吸収する構造**を設計 *[4]
- **非エンジニアへのAI活用展開** — CSは調査依頼時の仕様自己解決、デザイナー/マネージャーはモバイル・ウェブの簡易改修を自走できる環境を整備し、エンジニアの稼働を本質課題へ集中させる構造を実現

**成果**
- **6ヶ月間で全リポジトリ合計189件のAI関連PR**を創出
- **PR全体の約36%が size XS と判定され Auto Merge で完了**、チーム全体で**年間約86時間**のレビュー時間削減（20 PR/日 × 3分/PR × 240営業日換算）
- 非エンジニア職種が自走できる環境を整備し、エンジニア以外も含めた**組織全体の開発生産性向上**に寄与
- 他部署へナレッジ展開し、minne外へのAI活用拡大に貢献

#### 7. チーム・組織への貢献

- チーム横断の週次ミーティングを主催し、スプリントゴール達成率を**50% → 90%**に改善
- 新卒・中途メンバーのオンボーディングパートナーとしてGraphQL・Next.js・Storybook等の技術領域で成長を支援
- 通算**55件のインシデント**でハンドラーとして対応。即日〜翌営業日での収束を実現
